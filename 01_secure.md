# 01_secure

`step1`의 `SecureServlet`과 `01_secure.jsp`, `WEB-INF/01_secure2.jsp`를 통해
JSP/Servlet에서 "보이는 경로"와 "직접 접근할 수 없는 경로"의 차이를 정리한다.

## 핵심 개념

- `@WebServlet("/secure")`
  - 브라우저는 `/secure`로 요청하지만, 실제 처리 주체는 서블릿이다.
- `forward`
  - 주소창은 바뀌지 않고, 서버 내부에서 다음 자원으로 요청을 넘긴다.
- `WEB-INF`
  - 웹 애플리케이션 내부 전용 디렉터리다.
  - 외부에서 URL로 직접 접근할 수 없다.
- `request.setAttribute("isSecure", true)`
  - 서블릿이 만든 데이터를 JSP에 전달한다.

## 초심자를 위한 비유

이 구조는 "건물 로비"와 "직원 전용 공간"으로 생각하면 이해하기 쉽다.

- `01_secure.jsp`는 로비에 있는 안내판과 같다.
  - 누구나 찾아올 수 있는 공개 공간이다.
  - 파일이 `webapp` 아래에 있으면 보통 브라우저 요청으로 접근 가능하다.
- `WEB-INF/01_secure2.jsp`는 직원 전용 사무실과 같다.
  - 건물 안에서는 갈 수 있지만, 밖에서 문 주소를 직접 알아도 들어갈 수 없다.
  - 서블릿이 내부적으로 데려가야만 볼 수 있다.
- `SecureServlet`은 안내 직원과 같다.
  - 손님이 `"/secure"`라고 말하면, 내부 규칙에 따라 알맞은 방으로 안내한다.
  - 손님은 이동 사실을 체감하지 못한다. 주소창이 그대로이기 때문이다.

즉, "공개 가능한 페이지"와 "서버 내부에서만 쓰는 페이지"를 분리하는 것이 이 예제의 핵심이다.

## 능숙자를 위한 절차 설명

### 1. 요청 진입

- 클라이언트가 `GET /secure` 요청을 보낸다.
- `SecureServlet#doGet`이 매핑되어 요청을 받는다.

### 2. 요청 범위 데이터 세팅

- `req.setAttribute("isSecure", true)`를 통해 요청 객체에 데이터를 넣는다.
- 이 값은 같은 요청 흐름 안에서만 유효하다.

### 3. 내부 포워딩

- `req.getRequestDispatcher("/WEB-INF/01_secure2.jsp").forward(req, resp)`가 실행된다.
- 이 과정은 클라이언트 리다이렉트가 아니다.
- 따라서 다음 특징이 있다.
  - 브라우저 주소는 `/secure` 그대로 유지된다.
  - 새 요청이 아니라 같은 요청을 넘기는 방식이다.
  - request attribute를 그대로 사용할 수 있다.

### 4. JSP 렌더링

- `01_secure2.jsp`는 현재 예제 코드상 비어 있다.
- 실제로는 `request.getAttribute("isSecure")` 같은 값을 출력하거나,
  보안 여부에 따라 화면을 분기하는 용도로 사용할 수 있다.

## 이 예제가 보여주는 것

### 공개 자원과 보호 자원의 분리

- `src/main/webapp/01_secure.jsp`
  - 정적 자원처럼 외부 노출이 가능한 JSP다.
- `src/main/webapp/WEB-INF/01_secure2.jsp`
  - 외부 직접 접근이 차단된 JSP다.
  - 서블릿이 컨트롤러 역할을 해야만 접근할 수 있다.

### Forward와 Redirect의 차이

- `forward`
  - 서버 내부 이동이다.
  - 요청 데이터가 유지된다.
  - 주소창이 바뀌지 않는다.
- `redirect`
  - 클라이언트에게 새 URL로 다시 요청하라고 지시한다.
  - 요청이 새로 발생하므로 request attribute는 사라진다.
  - 주소창이 바뀐다.

이 예제는 `WEB-INF`와 `forward`를 함께 써서 "직접 접근 불가 + 내부 전달" 패턴을 보여준다.

## 면접 예상 문항

### 1. `WEB-INF` 아래에 JSP를 두는 이유는 무엇인가요?

- 외부 사용자가 URL로 직접 접근하지 못하게 하기 위해서다.
- 뷰 파일을 숨기고, 반드시 서블릿 같은 제어 계층을 거치게 만들 수 있다.

### 2. `forward`와 `redirect`의 차이를 설명해보세요.

- `forward`는 서버 내부에서 요청을 넘기며, 주소창이 변하지 않는다.
- `redirect`는 클라이언트에게 새 요청을 보내게 하며, 주소창이 바뀐다.
- `forward`는 같은 request를 유지할 수 있지만 `redirect`는 새 요청이므로 유지되지 않는다.

### 3. `request.setAttribute()`는 언제 사용하나요?

- 한 요청 안에서 서블릿이 만든 데이터를 JSP에 전달할 때 사용한다.
- 대표적으로 화면 표시용 값, 처리 결과, 검증 결과 등을 넘길 때 쓴다.

### 4. `SecureServlet`에서 `requestDispatcher("/WEB-INF/01_secure2.jsp")`를 사용한 이유는 무엇인가요?

- 직접 접근이 불가능한 JSP로 내부 포워딩하기 위해서다.
- 컨트롤러가 접근 제어를 담당하고, 뷰는 내부에서만 렌더링되도록 분리할 수 있다.

### 5. `01_secure.jsp`와 `WEB-INF/01_secure2.jsp`의 차이는 무엇인가요?

- `01_secure.jsp`는 외부에서 직접 접근 가능한 공개 JSP다.
- `WEB-INF/01_secure2.jsp`는 외부 직접 접근이 차단된 보호 JSP다.

### 6. 이 구조를 쓰면 어떤 장점이 있나요?

- 화면 자원을 직접 노출하지 않아 보안성이 좋아진다.
- 요청 처리와 화면 렌더링 책임을 나누기 쉬워진다.
- MVC 스타일 구조를 만들기 편해진다.

### 7. 주소창이 바뀌지 않는 이유는 무엇인가요?

- `forward`는 클라이언트 재요청이 아니라 서버 내부 전달이기 때문이다.

## 정리

- `SecureServlet`은 요청을 받아 보호된 JSP로 내부 전달한다.
- `WEB-INF`는 외부 직접 접근을 차단하는 핵심 장치다.
- `forward`는 주소창 변경 없이 request를 유지한 채 뷰로 이동한다.
- 이 예제는 "보여줄 화면"과 "직접 노출하면 안 되는 화면"을 나누는 기본 패턴을 학습하기에 좋다.
